# Фронтенд — MyKHSU Backend

Бэкенд использует **единый frontend-бандл** в `web-build/`, который включает:

1. **MyKHSU-web** — основной пользовательский интерфейс (React SPA)
2. **admin-panel** — панель администратора (React SPA)

---

## MyKHSU-web

Репозиторий: [PRO100BYTE/MyKHSU-web](https://github.com/PRO100BYTE/MyKHSU-web)

В этом репозитории MyKHSU-web подключён как git submodule в папке `mykhsu-web/`.

После клонирования backend-репозитория обязательно выполните:

```bash
git submodule update --init --recursive
```

### Подключение

```bash
# Автоматическая сборка объединенного фронтенда
npm run build:web

# Или вручную с указанием пути
node scripts/build-frontend.js /path/to/MyKHSU-web
```

По умолчанию скрипт использует сабмодуль `./mykhsu-web`.

Для пользовательской части «Единого окна» используется ветка сабмодуля `2.3.3-beta`.

После сборки:

- MyKHSU-web копируется в `web-build/`
- Admin Panel копируется в `web-build/admin-panel/`

В MyKHSU-web добавлена отдельная вкладка «Единое окно» с формой обращения (`visitor/student/teacher`) и отправкой на `POST /api/unified-window/tickets`.

История обращений в MyKHSU-web теперь берётся из БД через `GET /api/unified-window/tickets?contact_email=...` (по email), а не из browser storage.

Обе SPA раздаются одним Express-сервером из общей директории `web-build/`.

Для единой сборки всего проекта используйте:

```bash
npm run build
```

### Настройка proxy в dev-режиме

В `MyKHSU-web/package.json` должно быть:

```json
{
  "proxy": "http://localhost:8080"
}
```

Тогда при `npm start` из папки MyKHSU-web запросы `/api/*` проксируются на бэкенд.

### Что нужно поменять в MyKHSU-web

Файл `src/utils/constants.js` — убедитесь, что `API_BASE_URL` указывает на бэкенд:

```js
export const API_BASE_URL = process.env.NODE_ENV === 'development'
  ? '/api'                   // CRA proxy → localhost:8080
  : 'https://t2iti.khsu.ru/api';  // продакшн
```

### Кастомный API-эндпоинт (режим разработчика)

### Исправления и совместимость

- **Дневной режим расписания:** использует правильную синхронизацию номера недели через якорь `_serverWeekNumber`, что обеспечивает корректный расчёт дат при переходе между днями.
- **Поиск преподавателей и аудиторий:** фильтрует результаты от `/api/search/{query}` по типу (`teacher` или `auditory`), соответствуя формату API.
- **Расчёт дат:** аналогичен оригинальному мобильному приложению MyKHSU — использует понедельник текущей недели как якорь и смещение от текущей недели.

### Кастомный API-эндпоинт (режим разработчика)

В режиме разработчика MyKHSU-web позволяет переключаться на кастомный API-эндпоинт.

**Как активировать режим разработчика:**
1. Откройте **Настройки → Тема**
2. Кликните 7 раз на **"Акцентный цвет"** (или другом элементе UI)
3. Появится вкладка **Режим разработчика**

**Как переключиться на кастомный эндпоинт:**
1. В **Режиме разработчика** найдите раздел **"Кастомный API"**
2. Введите URL (например, `http://localhost:3000/api`)
3. Включите **"Использовать кастомный API"**
4. Нажмите **"Сохранить"** — приложение перезагружается и кэш очищается

**Особенности:**
- При смене эндпоинта автоматически очищается кэш (старое расписание не будет показано)
- При ошибке подключения приложение попытается использовать CORS-прокси
- Убедитесь, что кастомный API имеет правильные CORS-заголовки

## Admin Panel

Встроена в репозиторий в папке `admin-panel/`.

### Технологии

- **React 18** + React Router 7
- **Ionicons 7** (через CDN)
- **Шрифт Montserrat** (Google Fonts)
- CSS-переменные в стилистике MyKHSU (glass-morphism, dark/light theme)

### Сборка

```bash
npm run build:admin
# Эквивалентно:
cd admin-panel && npm install && npm run build
```

Собранные файлы админки сначала создаются в `admin-panel/build/`, затем автоматически копируются в `web-build/admin-panel/`.

В production используется путь `/admin-panel/` из общего frontend bundle.

### Доступные экраны

| Маршрут | Экран | Описание |
| --- | --- | --- |
| `/admin-panel/` | — | Редирект на /dashboard |
| `/admin-panel/login` | LoginScreen | Вход по логину и паролю |
| `/admin-panel/dashboard` | DashboardScreen | Ролевая сводка по разделам (поддержка/новости/расписание/пользователи), quick-actions по правам |
| `/admin-panel/schedule` | ScheduleScreen | Загрузка / очистка / ручное редактирование расписания по неделе / управление группами |
| `/admin-panel/times` | TimesScreen | Редактор расписания звонков |
| `/admin-panel/news` | NewsScreen | Управление новостями |
| `/admin-panel/users` | UsersScreen | Пользователи админки (CRUD + disable/enable) |
| `/admin-panel/login-history` | LoginHistoryScreen | История входов администраторов со всеми деталями (IP, ОС, браузер, устройство, пагинация, фильтры) |
| `/admin-panel/unified-window` | UnifiedWindowScreen | Обращения «Единого окна» и обработка статусов |
| `/admin-panel/profile` | ProfileScreen | Профиль текущего пользователя (имя, отчество, фамилия, должность, email, пароль) + история входов |
| `/admin-panel/appearance` | AppearanceScreen | Настройки темы и акцентного цвета в стиле MyKHSU |
| `/admin-panel/system-info` | SystemInfoScreen | Информация о системе (версии, git hash, сборка GMT+7, оперативность и runtime/DB статистика) |

### Версия/билд в интерфейсе

Версия, git hash и дата билда берутся из `GET /api/meta`.  
Резервные значения вынесены в `admin-panel/src/constants.js`.

При production-сборке (`npm run build`, `npm run build:web`, `npm run build:admin`) автоматически запускается `scripts/sync-build-info.js`: для admin/backend он подставляет версии из `src/constants.js`, актуальный git hash и дату/время сборки в часовом поясе Красноярска (`Asia/Krasnoyarsk`, GMT+7). Для MyKHSU-web версия приложения остается ручной, а автоматически обновляются только build-хеш и дата сборки в формате `DD.MM.YYYY` (как в предыдущем формате web-интерфейса).

### Темы

Admin Panel теперь синхронизирована с визуальной системой мобильного MyKHSU и поддерживает темы:

- `dark`
- `light`
- `matrix`
- `legend`

Текущая тема сохраняется в `localStorage` (ключ `admin_theme`), акцентный цвет — в `admin_accent`.

Пункт «Внешний вид» убран из основной боковой навигации и перенесён во всплывающее профильное меню блока пользователя внизу sidebar.

Кнопка сворачивания/разворачивания сайдбара перенесена вниз, над блоком профиля, чтобы не деформировать логотип в шапке.

Профильное меню (Профиль / Внешний вид / Выйти) открывается по клику и автоматически закрывается при клике вне меню.

В профильное меню добавлен отдельный пункт «Информация о системе», который открывает экран `/admin-panel/system-info`.

При свернутом sidebar профильное меню открывается как отдельный floating-popover и больше не наследует ширину боковой навигации.

В профильном меню доступны:

- редактирование профиля (логин, имя, фамилия, должность, email, смена пароля);
- быстрый переход к экрану оформления;
- выход из аккаунта.

На экране пользователей (`/admin-panel/users`) в форме создания/редактирования доступны поля: имя, отчество, фамилия, должность, email.

Эти же данные отображаются в таблице пользователей и в профиле текущего администратора (`/admin-panel/profile`).

Данные профиля загружаются/обновляются через `GET/PATCH /adminapi/profile` и хранятся в таблице `users` в `users.sqlite`.

Дополнительно на экране `/admin-panel/appearance` доступны:

- режим плотности интерфейса: `comfortable` / `compact` (`admin_ui_density`)

### Адаптивный макет Admin Panel

Админ-панель поддерживает три ключевых диапазона ширины:

- **Телефоны (`<= 768px`)**
- **Планшеты (`769px - 1024px`)**
- **ПК (`>= 1025px`)**

Ключевые особенности адаптива:

- на телефонах sidebar работает как **drawer** (выезжающее меню) с затемняющим backdrop;
- при навигации на мобильных устройствах drawer автоматически закрывается;
- в шапке на телефонах появляется отдельная кнопка открытия/закрытия меню;
- статистические блоки дашборда на телефонах работают как **горизонтальная snap-лента карточек**;
- при первом входе на мобильный дашборд показывается подсказка «Потяни карточки статистики вбок»;
- у мобильной snap-ленты статистики добавлены fade-края, визуально подсказывающие горизонтальный скролл;
- карточки и hero-блоки уменьшают внутренние отступы и радиусы на малых экранах;
- таблицы и формы переходят в более компактную типографику и интервалы;
- на планшетах уменьшены отступы layout и ширина sidebar для лучшей плотности контента;
- на широких экранах контентные блоки используют доступную ширину рабочей области (без фиксированного `max-width` для `screen-stack`).

### Акцентные цвета

Доступны акценты, соответствующие мобильному приложению:

- `green`
- `blue`
- `purple`
- `orange`
- `matrix`
- `legend`

Токены вынесены в `admin-panel/src/constants.js`.

### История входов

#### Профиль пользователя (локальная история)

На странице профиля (`/admin-panel/profile`) отображаются последние 20 входов текущего пользователя:

- Дата и время входа
- IP адрес
- Операционная система (ОС) и версия
- Браузер и версия
- Модель устройства

Информация об устройстве парсится из User-Agent при помощи `ua-parser-js`.

API эндпоинт: `GET /adminapi/users/{id}/login-history?limit=50&offset=0`

#### История входов для администратора

Отдельный экран `/admin-panel/login-history` доступен администраторам с правом `users:read`.

Функциональность:

- Таблица со всеми входами всех пользователей
- Фильтр по ID пользователя
- Сортировка (новые сверху / старые сверху)
- Пагинация (100 записей на странице, максимум 500)
- Отображение: дата/время, пользователь (логин + ФИО), IP, ОС, браузер, модель устройства

API эндпоинты:
- `GET /adminapi/login-history?limit=100&offset=0&userId={id}` — таблица входов
- `GET /adminapi/users/{id}/login-stats` — статистика (всего входов, уникальные дни/IP, список ОС/браузеров)

### Дизайн

Стилистика синхронизирована с React Native приложением MyKHSU:

- токены `LIQUID_GLASS` и палитра акцентов перенесены в desktop-админку
- используется фирменная бренд-марка из девяти плиток вместо условной буквы-логотипа
- фон построен на тех же радиальных свечениях green/purple/blue, что и mobile-макеты
- навигация и элементы управления оформлены как pill/glass-сегменты
- темы `matrix` и `legend` повторяют специальные режимы мобильного приложения
- build/version-метаданные выводятся в интерфейс через `GET /api/meta`
- в блоке системной информации дополнительно выводятся: git hash, timezone сборки, runtime Node.js (uptime, память) и объёмы ключевых сущностей БД
- добавлен отдельный экран внешнего вида (theme/accent preview), аналогичный mobile-подходу
- добавлены UX-настройки навигации и плотности интерфейса (desktop-аналог mobile Appearance Settings)
- экран входа переработан: полноценная анимация карточки/блоков, аккуратная форма и строка-подсказка «Забыли пароль?» с glass-tooltip
- tooltip «Забыли пароль?» отображается поверх формы входа и не обрезается контейнером карточки
- на экране входа используется только бренд-марка приложения (без служебного значка-щита), подписи «Логин/Пароль» и их иконки располагаются в одной строке над полями ввода
- логотип на форме входа визуально унифицирован с бренд-блоком авторизованной админ-панели
- авторизация в админке использует всплывающие glass-toast уведомления: успех входа, неверные учетные данные, отключенная учетная запись, ошибка загрузки ролей, сетевые и неизвестные ошибки
- в ошибочных уведомлениях админки показывается короткий код ошибки для быстрой идентификации (`ADM-AUTH-00X`, `UI-AUTH-00X`, `UI-NET-001`)
- в экранах расписания, новостей, звонков и пользователей служебные уведомления унифицированы через единый toast-слой вместо разрозненных inline-alert блоков
- экран `/admin-panel/unified-window` переведен в чат-режим: список обращений + переписка, sticky-шапка и sticky-композер, прокрутка только в области сообщений
- в админском чате добавлена явная кнопка отправки сообщения и отправка по `Enter` (`Shift+Enter` — перенос строки)
- смена статуса обращения вынесена в отдельное модальное окно
- добавлено удаление обращения в админке с обязательным подтверждением
- в списке обращений админки выводятся номер, тема, имя, тип пользователя, email, источник последнего сообщения и индикатор непрочитанных ответов
- в админском экране обращений зафиксирован выбор тикета при обновлении списка (перечитывание не сбрасывает открытый диалог)
- исправлен сбой рендера экрана «Единое окно» в админке (пустой фон без UI) из-за некорректной ссылки на идентификатор обращения при обновлении списка
- quick-actions на `/admin-panel/dashboard` автоматически скрывают недоступные разделы в зависимости от прав текущего пользователя
- дашборд теперь формирует сводку по роли пользователя:
  - агент поддержки: обращения, неотвеченные, непрочитанные, статусы
  - редактор новостей: количество и частота публикаций
  - диспетчер расписания: заполненные недели, курсы, группы по курсам, граница заполнения
  - менеджер: агрегированная сводка по расписанию/новостям/обращениям
  - администратор: полная сводка, распределение пользователей по ролям, свежесть разделов
- toast-уведомления в админке анимируются и при появлении, и при закрытии
- профильное меню (в блоке пользователя sidebar) анимируется при открытии и закрытии
- модальные окна в админке (новости, пользователи, смена статуса обращений, confirm-диалоги) анимируются и при открытии, и при закрытии
- подтверждения опасных действий в админке выполняются через кастомное модальное окно в стиле панели (без браузерных `alert/confirm`)
- в разделе расписания ручной ввод и редактирование объединены в режим «Ручное редактирование»: выбор курс/группа/неделя и табличное заполнение недели
- в режиме «Ручное редактирование» расписание на выбранные курс/группу/неделю автоматически подгружается в сетку (дни недели × пары); при отсутствии данных отображается пустая таблица для создания расписания
- в заголовках дней режима «Ручное редактирование» отображаются календарные даты (по выбранной неделе)
- сохранение в табличном режиме выполняет дифф по ячейкам: существующие пары обновляются/удаляются, новые пары создаются
- в ручном редакторе расписания добавлен экспорт выбранной комбинации курс/группа/неделя в JSON (`Timetable`-совместимый формат)
- в ручном редакторе добавлено копирование расписания из выбранной недели/группы в целевую неделю и/или целевую группу (с опцией замены существующих пар)
- фильтры ручного редактирования (курс/группа/неделя) вынесены в отдельные подписанные поля с более компактными селекторами для удобной работы
- в разделе расписания добавлена отдельная вкладка «Управление группами» (создание, переименование и удаление групп по выбранному курсу)
- в «Управлении группами» добавлено удаление выбранного курса (вместе с группами этого курса в каталоге) с подтверждением опасного действия
- MyKHSU-web запрашивает расписание, курсы, группы, недели и звонки без долгоживущего кэша, чтобы изменения после импорта в админке отображались сразу
- очистка расписания подтверждается через модальное окно confirm-диалога
- в разделе звонков сохранение изменений и удаление строки подтверждаются через модальное окно confirm-диалога
- в разделе звонков добавлена кнопка «Отменить последнее» для отката последнего действия до сохранения
- исправлена ошибка загрузки сводки на дашборде, вызванная SQL-условием с некорректной проверкой пустой строки
- в профиле и управлении пользователями добавлено необязательное поле отчества (`middle_name`)
- для пунктов sidebar добавлены дополнительные интервалы между элементами, чтобы hover/active-обводки не конфликтовали визуально
- в Едином окне добавлена проверка по словарю нецензурной лексики (RU/EN):
  - при создании обращения в MyKHSU-web проверяются поля имя/email/тема/сообщение и выводится ошибка с перечислением затронутых полей
  - при отправке сообщения в чате MyKHSU-web и админ-панели сообщение блокируется с уведомлением о необходимости использовать корректные выражения
- при создании обращения на фронтенде роль заявителя передается в совместимом формате (`role` и `requester_role`) для корректного отображения типа пользователя в админке

Для пользовательского интерфейса «Единого окна» (MyKHSU-web):

- стартовый экран предлагает выбор: создать новое обращение или найти существующие по email
- при создании обращения поле email обязательно (нужно для последующего поиска обращений)
- в чате добавлено закрытие обращения пользователем с обязательным комментарием
- в чате зафиксированы шапка и нижняя панель ввода, прокручивается только история сообщений
- в списке обращений MyKHSU-web добавлены поля: номер, тема, имя, тип пользователя, email, источник последнего сообщения
- на MyKHSU-web пользователю разрешено удаление закрытых и решенных обращений с подтверждением
- кнопка удаления обращения перенесена из списка обращений в само окно обращения и показывается под заглушкой для статусов «Закрыто/Решено»
- при переоткрытии обращения агентом (из «Решено/Закрыто» в «Открыто/В работе») ранее скрытое пользователем обращение снова появляется на фронтенде
- для пользователя и агента реализованы индикаторы непрочитанных ответов в списках обращений
- экран «Поиск обращений по email» в MyKHSU-web вынесен в отдельный сценарий (по аналогии с формой создания): только поле email и кнопка «Найти обращения», без кнопок создания обращений
- если по указанному email обращений нет, в MyKHSU-web показывается отдельная заглушка «По указанному email обращений не найдено»
- элементы интерфейса «Единого окна» в MyKHSU-web принудительно используют шрифт Montserrat для визуальной консистентности с остальным приложением

## Маршруты в Express

```text
GET  /               → web-build/index.html (MyKHSU-web SPA)
GET  /admin-panel/*  → web-build/admin-panel/index.html (Admin SPA)
GET  /api/*          → Публичный API
POST /adminapi/*     → Административный API
GET  /static/*       → legacy статика (из STATIC_PATH)
```

Для диагностики ошибок API сервер пишет подробные записи по каждому запросу и runtime-ошибке в отдельный файл `logs/server.log`.

## Разработка Admin Panel

```bash
cd admin-panel
npm install
npm start   # http://localhost:3000 (прокси на localhost:8080)
```

При разработке CRA проксирует `/adminapi/*` и `/api/*` на бэкенд (настройка `proxy` в `admin-panel/package.json`).
